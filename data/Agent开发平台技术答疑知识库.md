

### 1. 什么是工作流（Workflow）和对话流（Chatflow）？它们有什么区别？

**工作流（Workflow）** 是一系列可执行指令的集合，用于实现业务逻辑或完成特定任务。它为智能体的数据流动和任务处理提供了一个结构化框架，核心在于将大模型的能力与特定业务逻辑相结合，通过系统化、流程化的方法实现高效、可扩展的AI应用开发。

**对话流（Chatflow）** 是基于对话场景的特殊工作流，更适合处理对话类请求。它通过对话的方式和用户交互，并完成复杂的业务逻辑。

两者的核心区别：
- **适用场景不同**：工作流适合数据自动化处理场景，如生成行业调研报告、生成海报、制作绘本等；对话流适用于Chatbot等需要在响应请求时进行复杂逻辑处理的对话式应用程序，如个人助手、智能客服、虚拟伴侣等。
- **交互方式不同**：工作流是单次输入→处理→输出的模式；对话流支持多轮对话，可以在执行过程中与用户进行交互确认。
- **节点类型差异**：对话流特有"用户问题"节点，可以在流程中间暂停等待用户输入；工作流则是一次性执行完毕。

### 2. 工作流中有哪些常用节点类型？各自的作用是什么？

Coze平台的工作流画布提供以下核心节点类型：

| 节点类型 | 功能说明 |
|---------|---------|
| **开始节点** | 定义工作流的输入参数，是工作流的入口 |
| **结束节点** | 定义工作流的输出，是工作流的出口 |
| **大模型节点（LLM）** | 调用大语言模型进行文本生成、理解、推理等任务 |
| **代码节点** | 执行自定义代码（Python/JavaScript），用于数据处理和逻辑判断 |
| **条件判断节点（IF）** | 根据条件表达式进行分支判断，实现不同流程走向 |
| **插件节点** | 调用已发布的插件工具，扩展智能体能力 |
| **知识库节点** | 从知识库中检索相关内容，增强大模型的回答准确性 |
| **变量节点** | 读取或写入变量，用于节点间数据传递 |
| **消息节点** | 向用户发送中间消息，常用于对话流中展示中间结果 |
| **循环节点** | 对列表数据进行遍历处理，支持批量操作 |
| **子工作流节点** | 调用其他已发布的工作流，实现工作流的嵌套复用 |
| **图像生成节点** | 调用文生图模型生成图片 |
| **HTTP请求节点** | 发送HTTP请求调用外部API服务 |
| **数据库节点** | 对数据库进行增删改查操作 |

### 3. 如何在工作流中实现节点之间的数据传递？

工作流中的数据传递主要通过**变量系统**实现，核心机制如下：

**变量类型：**
- **输入变量**：工作流开始节点定义的参数，作为整个工作流的入口数据
- **节点输出变量**：每个节点执行后产生的结果，可被下游节点引用
- **系统变量**：平台内置的变量，如当前时间、用户信息等

**引用方式：**
在节点的输入参数中，使用 `{{变量名}}` 语法引用上游节点的输出。例如：
- 引用开始节点的输入：`{{input}}`
- 引用上一个LLM节点的输出：`{{LLM_node.output}}`
- 引用代码节点的返回值：`{{Code_node.result}}`

**最佳实践：**
1. 每个节点建议重命名，使用有意义的名称（如"用户意图识别"、"生成回复"），方便后续引用
2. 代码节点需要明确定义输出变量的结构
3. 条件判断节点的各分支可以设置不同的输出
4. 使用变量聚合节点可以合并多个分支的输出

### 4. 条件判断节点如何配置？支持哪些判断方式？

条件判断节点用于根据不同的条件执行不同的分支逻辑，支持以下配置方式：

**判断模式：**
- **IF/ELSE模式**：最基本的二分支判断，满足条件走IF分支，否则走ELSE分支
- **多条件分支**：支持设置多个条件，按优先级从上到下匹配，匹配成功即执行对应分支
- **表达式判断**：支持使用JavaScript表达式进行条件判断

**支持的判断类型：**
- 字符串：包含、不包含、等于、不等于、正则匹配
- 数字：大于、小于、等于、大于等于、小于等于
- 布尔值：为真、为假
- 列表：为空、不为空、包含、不包含
- 变量：是否存在、是否为空

**配置示例：**
```
条件1: {{user_intent}} == "查询天气"  → 执行天气查询分支
条件2: {{user_intent}} == "设置提醒"  → 执行提醒设置分支
默认: ELSE → 执行兜底回复分支
```

### 5. 子工作流（Sub-workflow）如何使用？有什么注意事项？

子工作流是将一个完整的工作流作为节点嵌入到另一个工作流中使用，实现模块化和复用。

**使用步骤：**
1. 先创建并发布要作为子工作流的工作流
2. 在父工作流中添加"子工作流"节点
3. 在节点配置中选择已发布的工作流
4. 配置子工作流的输入参数（数据来源可以是固定值或引用上游节点输出）
5. 子工作流的输出可被父工作流的后续节点引用

**注意事项：**
- 子工作流发布新版本后，引用它的父工作流**不会自动更新**到最新版本，以免影响线上业务
- 更新子工作流后，需要手动更新父工作流中的子工作流版本
- 子工作流支持**批处理模式**，可以对列表数据逐项处理
- 避免循环嵌套（A调用B，B又调用A），否则会导致死循环
- 子工作流的输入输出结构由其自身定义，不支持在父工作流中自定义

### 6. 工作流调试有哪些方法？如何快速定位问题？

Coze平台提供了完善的调试工具：

**调试方法：**
1. **试运行（Test Run）**：在画布右上角点击"试运行"，手动输入测试数据执行整个工作流
2. **单节点调试**：右键某个节点选择"试运行此节点"，只执行该节点及其上游依赖
3. **断点调试**：在对话流中可以设置断点，暂停在特定节点查看中间状态
4. **日志查看**：每个节点执行后可以查看输入、输出、耗时、token消耗等详细信息

**常见问题定位：**
- **节点报错**：查看节点的错误信息，通常是参数格式不对或API调用失败
- **输出为空**：检查上游节点是否正确输出，变量引用路径是否正确
- **条件分支走错**：检查条件表达式，确认变量的实际值
- **超时**：检查是否有节点执行时间过长，考虑增加超时设置或优化逻辑
- **变量未定义**：确认变量名拼写正确，且上游节点已正确输出该变量

### 7. 工作流中的批处理（Batch）是什么？如何使用？

批处理是指对输入的列表数据逐项执行相同的操作，类似于编程中的for循环。

**使用场景：**
- 批量处理用户上传的多张图片
- 对一组关键词逐一进行搜索和分析
- 批量发送消息给多个用户

**配置方式：**
1. 在支持批处理的节点（如LLM节点、插件节点）中开启"批处理"开关
2. 设置批处理的输入数组（必须是列表类型）
3. 配置批处理的并发数（控制同时执行的数量）
4. 节点会对数组中的每个元素执行一次，输出结果也是一个列表

**注意事项：**
- 批处理会消耗更多token和API调用次数
- 建议设置合理的并发数，避免触发API限流
- 批处理节点的输出是数组，后续节点需要按索引访问或使用循环节点处理

## 二、提示词调优（7题）

### 8. 什么是系统提示词和用户提示词？如何合理搭配使用？

**系统提示词（System Prompt）：**
是给大模型的"人设"和"行为准则"，定义了智能体的身份、能力边界和回复风格。用户不可见，但对模型行为有决定性影响。

**用户提示词（User Prompt）：**
是用户实际输入的问题或指令，是模型需要处理的具体任务。

**搭配使用的最佳实践：**

系统提示词应包含：
1. **角色定义**：你是一个专业的XX助手
2. **能力说明**：你可以帮助用户做XX、XX
3. **行为约束**：不要做XX、遇到XX情况应该XX
4. **输出格式**：请以XX格式回复
5. **知识背景**：相关的背景信息和专业知识

用户提示词应：
1. 简洁明确地表达需求
2. 必要时提供上下文信息
3. 明确期望的输出格式

**示例：**
```
【系统提示词】
你是一个专业的法律顾问助手，擅长解答劳动法相关问题。
回答时需要引用具体的法律条文，并用通俗易懂的语言解释。
不要给出具体的诉讼建议，建议用户咨询专业律师。

【用户提示词】
公司拖欠工资3个月，我该怎么办？
```

### 9. 如何编写高质量的系统提示词？有哪些技巧？

**提示词编写的核心原则：**

1. **明确角色定位**
   - ✅ "你是一个专注于Python后端开发的技术顾问"
   - ❌ "你是一个AI助手"

2. **具体而非笼统**
   - ✅ "回答控制在200字以内，使用Markdown格式，包含代码示例"
   - ❌ "回答要简洁"

3. **提供示例（Few-shot）**
   ```
   用户问：什么是装饰器？
   回答格式：
   ## 概念解释
   [简要说明]
   ## 代码示例
   [代码块]
   ## 使用场景
   [列举2-3个]
   ```

4. **设置边界和约束**
   - 明确说明不能做什么
   - 设置拒绝话术模板
   - 定义兜底回复

5. **使用分隔符**
   - 用`---`、`###`等分隔不同部分
   - 用XML标签包裹特殊内容，如`<context>背景信息</context>`

6. **链式思考（Chain of Thought）**
   - 对于复杂任务，引导模型"一步步思考"
   - "请先分析问题，然后给出解决方案，最后总结"

### 10. 提示词中的变量如何使用？有什么作用？

Coze平台支持在提示词中使用变量，实现动态化和个性化。

**变量类型：**
- `{{user_input}}`：用户的原始输入
- `{{conversation_history}}`：对话历史
- `{{knowledge}}`：知识库检索结果
- `{{current_time}}`：当前时间
- 自定义变量：通过变量节点设置的自定义值

**使用场景：**
1. **个性化回复**：在提示词中引用用户信息变量
   ```
   你好{{user_name}}，我是你的专属助手...
   ```

2. **动态上下文**：引用知识库检索结果
   ```
   根据以下参考资料回答用户问题：
   {{knowledge}}
   
   用户问题：{{user_input}}
   ```

3. **条件化提示**：根据变量值改变提示词内容
   ```
   {% if language == "en" %}
   Please answer in English.
   {% else %}
   请用中文回答。
   {% endif %}
   ```

**注意事项：**
- 变量名必须与上游节点定义的输出变量名完全一致
- 变量值过长时可能占用大量token，建议先进行摘要处理
- 使用Jinja2语法可以实现更复杂的变量处理逻辑

### 11. 如何优化提示词以减少大模型的"幻觉"问题？

大模型幻觉（Hallucination）是指模型生成看似合理但实际不正确的内容。以下是减少幻觉的有效方法：

**方法一：知识库增强（RAG）**
- 将专业知识文档导入知识库
- 在提示词中引用知识库检索结果
- 明确要求"根据提供的资料回答，不要编造"

**方法二：约束性提示词**
```
请严格根据以下规则回答：
1. 只回答你确定的信息
2. 如果不确定，明确说明"我不确定"
3. 不要编造不存在的数据或引用
4. 涉及具体数字时，请引用来源
```

**方法三：思维链验证**
```
请按以下步骤回答：
1. 先分析用户的问题
2. 从知识库中查找相关信息
3. 基于找到的信息组织答案
4. 最后检查答案是否有依据
```

**方法四：输出格式约束**
- 要求模型输出信息来源
- 使用结构化格式（如JSON）便于验证
- 添加"如果无法确认，请标注[待验证]"

### 12. 提示词太长会有什么影响？如何精简优化？

**提示词过长的影响：**
1. **token消耗增加**：每次对话都会消耗更多token，增加成本
2. **响应变慢**：模型处理更长的输入需要更多时间
3. **注意力分散**：模型可能忽略关键指令（"Lost in the Middle"问题）
4. **上下文窗口溢出**：加上对话历史，可能超出模型的最大上下文长度

**精简优化方法：**

1. **去除冗余信息**
   - 删除重复的指令
   - 合并相似的要求
   - 用简洁的语言表达

2. **使用知识库替代**
   - 将长篇背景资料放入知识库
   - 提示词中只保留核心指令
   - 通过RAG动态检索相关内容

3. **分层设计**
   - 基础提示词：核心角色和规则（精简）
   - 动态内容：通过变量注入上下文
   - 用户输入：实际问题

4. **使用模板和缩写**
   - 建立常用指令的简写
   - 使用提示词模板复用

**建议长度：** 系统提示词控制在500-1500字为宜，超过2000字建议拆分或使用知识库。

### 13. 如何进行提示词的A/B测试和迭代优化？

**测试流程：**

1. **准备测试用例**
   - 收集典型用户问题（20-50个）
   - 覆盖不同场景和难度
   - 标注期望的输出结果

2. **创建对比版本**
   - A版本：当前提示词
   - B版本：修改后的提示词
   - 每次只修改一个变量，便于归因

3. **评估维度**
   - 准确性：回答是否正确
   - 相关性：回答是否切题
   - 完整性：是否覆盖关键信息
   - 格式：是否符合要求的输出格式
   - 风格：语气和用词是否合适

4. **迭代优化**
   - 记录每次修改的内容和效果
   - 保留效果好的版本
   - 持续收集用户反馈进行优化

**Coze平台工具：**
- 使用"调试与预览"面板实时测试
- 利用"AI优化"功能自动改写提示词
- 通过"效果评测"功能批量测试

### 14. 多轮对话场景下，提示词设计需要注意什么？

**多轮对话的特殊挑战：**
- 上下文累积导致token消耗增加
- 模型可能忘记早期对话内容
- 需要维护对话状态和用户意图

**设计要点：**

1. **上下文管理**
   ```
   请根据对话历史理解用户的真实意图。
   如果用户使用代词（如"它"、"那个"），请根据上下文推断具体指代。
   对话历史：{{conversation_history}}
   ```

2. **状态追踪**
   - 使用变量记录对话状态（如已收集的信息、当前步骤）
   - 在提示词中说明当前状态
   ```
   当前对话状态：已收集用户的姓名和问题类型，下一步需要确认具体需求。
   ```

3. **摘要压缩**
   - 对于长对话，定期生成对话摘要
   - 用摘要替代完整的历史记录
   - 保留关键信息，丢弃冗余内容

4. **意图澄清**
   ```
   如果用户的意图不明确，请主动询问以澄清。
   不要假设用户的意图，而是通过提问确认。
   ```

5. **话题切换处理**
   ```
   如果用户突然切换话题，请：
   1. 确认是否完成了之前的话题
   2. 自然地过渡到新话题
   3. 不要混淆不同话题的上下文
   ```

## 三、工具调用（6题）

### 15. 什么是插件（Plugin）？插件和工具是什么关系？

**插件（Plugin）** 是一个工具集，一个插件内可以包含一个或多个工具（API）。插件用于拓展智能体的能力边界，让智能体能够执行大模型本身无法完成的任务。

**插件与工具的关系：**
- 一个插件 = 多个工具的集合
- 每个工具 = 一个独立的API接口
- 同一插件内的工具必须使用相同的域名

**示例：**
一个"天气服务"插件可能包含：
- 工具1：查询当前天气 → `http://api.weather.com/current`
- 工具2：查询未来天气 → `http://api.weather.com/forecast`
- 工具3：查询天气预警 → `http://api.weather.com/alert`

**插件的类型：**
1. **官方插件**：平台预置的插件，如搜索、天气、新闻等
2. **第三方插件**：其他开发者创建并发布的插件
3. **自定义插件**：用户自己创建的插件，集成私有API

### 16. 如何创建自定义插件？详细步骤是什么？

**创建自定义插件的步骤：**

**第一步：准备API服务**
- 确保你的API服务已部署并可访问
- 准备好API文档，包括请求方式、参数、返回格式
- 如需要，准备API Key或OAuth认证信息

**第二步：创建插件**
1. 进入Coze平台的"资源库" → "插件"
2. 点击"创建插件"
3. 填写插件名称、描述、图标
4. 选择认证方式（无需认证/API Key/OAuth）

**第三步：添加工具**
1. 在插件中点击"添加工具"
2. 填写工具名称和描述（描述很重要，AI根据它判断何时调用）
3. 配置API信息：
   - 请求方式：GET/POST/PUT/DELETE
   - 请求URL：完整的API地址
   - 请求参数：定义参数名、类型、是否必填、描述
   - 请求头：如需要添加认证信息
   - 请求体：POST请求的body结构

**第四步：调试与测试**
1. 使用"测试"功能验证工具是否正常工作
2. 检查参数传递是否正确
3. 验证返回结果格式

**第五步：发布**
1. 确认所有工具测试通过
2. 点击"发布"
3. 发布后即可在工作流或智能体中使用

### 17. 工具调用（Function Calling）的原理是什么？智能体如何决定调用哪个工具？

**Function Calling原理：**

1. **工具注册**：将所有可用工具的名称、描述、参数定义提供给大模型
2. **意图理解**：大模型分析用户输入，判断是否需要调用工具
3. **工具选择**：如果需要，模型根据工具描述选择最合适的工具
4. **参数生成**：模型根据用户输入生成工具调用所需的参数
5. **执行调用**：平台调用实际的API接口
6. **结果处理**：将API返回结果交给模型进行总结和回复

**模型如何选择工具：**
- 主要依据工具的**描述（description）**来判断
- 描述越准确、越具体，模型选择越精准
- 参数的描述也很重要，帮助模型正确填充参数

**优化工具描述的建议：**
```
✅ 好的描述：
"查询指定城市的当前天气信息，包括温度、湿度、风力等。当用户询问天气相关问题时使用。"

❌ 差的描述：
"天气查询"
```

### 18. 工具调用失败时如何处理？有哪些常见错误和解决方案？

**常见错误类型及解决方案：**

| 错误类型 | 原因 | 解决方案 |
|---------|------|---------|
| **参数错误** | 模型生成的参数格式不对 | 完善参数描述，添加示例；在代码节点中添加参数校验 |
| **认证失败** | API Key过期或无效 | 检查认证配置，更新API Key |
| **超时** | API响应太慢 | 增加超时设置；优化API性能；添加重试机制 |
| **限流** | 调用频率超过限制 | 降低调用频率；添加队列和延迟；升级API套餐 |
| **服务不可用** | 外部服务宕机 | 添加错误处理和兜底回复；监控服务状态 |
| **返回格式异常** | API返回格式变化 | 添加返回值校验；使用代码节点进行格式转换 |

**错误处理最佳实践：**

1. **在工作流中添加条件判断**
   - 检查工具调用是否成功
   - 失败时执行备用方案或给出友好提示

2. **设置重试机制**
   - 对于临时性错误（如超时、限流），自动重试
   - 设置最大重试次数，避免无限循环

3. **提供兜底回复**
   ```
   当工具调用失败时，回复：
   "抱歉，当前服务暂时不可用，请稍后再试。如有紧急需求，请联系人工客服。"
   ```

4. **记录错误日志**
   - 在代码节点中记录错误信息
   - 便于后续排查和优化

### 19. 如何在工作流中调用HTTP请求节点？支持哪些配置？

HTTP请求节点允许工作流直接调用外部API，无需创建插件。

**支持的配置项：**

1. **请求方法**：GET、POST、PUT、DELETE、PATCH
2. **请求URL**：支持使用变量动态拼接URL
3. **请求头（Headers）**：
   - Content-Type
   - Authorization（API Key、Bearer Token等）
   - 自定义Header
4. **查询参数（Query Params）**：URL参数，支持变量
5. **请求体（Body）**：
   - JSON格式
   - Form表单格式
   - 原始文本
6. **超时设置**：请求超时时间（默认30秒）
7. **重试策略**：失败后是否重试、重试次数

**使用示例：**

调用一个翻译API：
```
方法：POST
URL：https://api.translate.com/v1/translate
Headers：
  Content-Type: application/json
  Authorization: Bearer {{api_key}}
Body：
  {
    "text": "{{input_text}}",
    "from": "zh",
    "to": "en"
  }
```

**注意事项：**
- URL必须是公网可访问的地址
- 敏感信息（如API Key）建议使用变量，不要硬编码
- 返回结果通常是JSON格式，需要在后续节点中解析
- 部分API可能有CORS限制，但服务端调用不受影响

### 20. 智能体如何使用知识库（Knowledge Base）增强回答能力？

**知识库的作用：**
解决大模型专业领域知识不足和幻觉问题，通过检索增强生成（RAG）技术提升回答准确性。

**使用流程：**

**第一步：创建知识库**
1. 进入"资源库" → "知识库"
2. 创建新知识库，填写名称和描述
3. 选择知识库类型：
   - **文本知识库**：适合文档、FAQ等非结构化内容
   - **表格知识库**：适合结构化数据，如产品参数表

**第二步：上传数据**
- 支持多种数据源：本地文档（PDF/Word/TXT）、网页链接、Notion、飞书文档等
- 设置分段规则：按段落、按字符数、自定义分隔符
- 等待系统自动向量化处理

**第三步：在智能体/工作流中使用**
- **智能体配置**：在智能体设置中关联知识库，系统会自动检索
- **工作流节点**：添加"知识库"节点，手动配置检索参数

**知识库节点配置：**
- **检索模式**：
  - 自动检索：根据用户输入自动匹配相关内容
  - 关键词检索：使用指定关键词搜索
- **返回数量**：设置返回的内容片段数量（建议3-5条）
- **相似度阈值**：过滤低相关性的结果

**最佳实践：**
1. 知识库内容保持更新，定期维护
2. 合理设置分段长度，太长影响检索精度，太短丢失上下文
3. 在提示词中明确要求模型"根据知识库内容回答"
4. 对于高频问题，可以直接写入提示词而非依赖检索
